# 🛡️ PhishGuard — Explainable AI Phishing Email Detector

A machine-learning-powered web application and browser extension that detects phishing emails in real time. Built with **Flask**, **scikit-learn**, and **vanilla JavaScript**, it goes beyond a simple "safe/phishing" verdict by providing **explainable AI (XAI) word-weight heatmaps**, a **Hybrid Decision Fusion Pipeline** (combining ML text intent with absolute SPF/DKIM/institutional overrides), an **asynchronous Gmail/Outlook DOM scraper extension**, a **live OpenPhish threat intelligence feed**, and a **tabbed manual scanner** — all supporting dynamic light/dark theme adaptation.

## 🔬 How It Works — The Hybrid Decision Fusion Pipeline

Every email analyzed is evaluated through a hybrid model that uses the Machine Learning model's text intent prediction as a baseline and layers heuristic rules as absolute overrides:

```
                        ┌──────────────────────────────┐
                        │       Email Text Input        │
                        └──────────────┬───────────────┘
                                       │
          ┌────────────────────────────┼────────────────────────────┐
          │                            │                            │
          ▼                            ▼                            ▼
  ┌───────────────┐          ┌──────────────────┐         ┌──────────────────┐
  │ 1. ML Model   │          │ 2. Metadata      │         │ 3. Rule-Based    │
  │               │          │    Analysis      │         │    Risk Score    │
  │ TF-IDF →      │          │                  │         │                  │
  │ Classifier    │          │ Sender domain    │         │ Suspicious term  │
  │ (NB/LR/RF)    │          │ DKIM header      │         │ frequency count  │
  │               │          │ Attachment ext   │         │ → 0-100 score    │
  │ → label +     │          │ URL patterns     │         │ → risk level     │
  │   confidence  │          │ OpenPhish feed   │         │ → action         │
  └───────┬───────┘          └────────┬─────────┘         └──────────┬───────┘
          │                           │                              │
          └─────────────┐             │                              │
                        ▼             ▼                              │
               ┌─────────────────────────────┐                      │
               │  4. XAI — Explain Prediction│                      │
               │                             │                      │
               │  Per-word feature weights   │                      │
               │  computed from pipeline     │                      │
               │  internals (top 15 words)   │                      │
               └─────────────┬───────────────┘                      │
                             │                                       │
                             ▼                                       ▼
                  ┌──────────────────────────────────────────────────────┐
                  │                JSON Response to Browser              │
                  │  prediction · confidence · model_used · sender ·     │
                  │  domain_spoof · is_verified_brand · is_dkim_signed · │
                  │  attachment_risk · has_url · suspicious_url ·        │
                  │  risk_score · risk_level · recommended_action ·      │
                  │  xai_weights · stats counters                        │
                  └──────────────────────────────────────────────────────┘
```

### Decision Fusion Override Logic

1. **Absolute Trust Override:** If the domain is an authenticated official institutional root (`.gov`, `.gov.in`, `.nic.in`, `.edu`, `.edu.in`, `.ac.in`) or an aligned corporate brand domain, and no spoof is detected, the prediction is **forced to Safe** (risk score capped at 15%) regardless of text suspicion.
2. **Critical Threat Overrides:** The verdict is **forced to Phishing / Dangerous** (risk score forced to >= 80% / 95%) if any of the following are met:
   * **Domain Spoofing:** Mismatched senders mimicking well-known brands (typosquatting).
   * **Risky Attachments:** High-risk executables (`.exe`, `.bat`) that haven't been marked as clean by Google Antivirus.
   * **Raw IP Links:** Any URL containing a raw numerical IP address (e.g., `http://192.168.1.1`).
   * **OpenPhish Listed:** Domain is actively listed in the OpenPhish database of verified phishing scams.
3. **ML Baseline:** If no overrides trigger, the system defaults to the output of the active TF-IDF machine learning pipeline.

## 🏋️ Training the Model

Our model is trained on a high-quality **18,642-row Kaggle Phishing Email dataset** augmented with custom Indian transactional Ham templates (safe NEET registration alerts, IFFCO platform logins, Google mail delivery failure bounces, and OTP codes) to eliminate false positives.

To download the dataset and execute the retraining script:
```bash
python download_and_train.py
```
This downloads the CSV, preprocesses column names, merges our templates, and serializes the trained models (`model.pkl`) with **96.94% accuracy** (Logistic Regression).

## 🔌 Chrome/Firefox Extension Integration

The Chrome/Firefox browser extension is located in the `/extension` directory.

### How to Install:
1. Open Chrome and navigate to `chrome://extensions/`.
2. Enable **Developer mode** (top-right toggle).
3. Click **Load unpacked** (top-left) and select the `extension` folder inside this project directory.
4. Open **Gmail** or **Outlook** in your browser, open an email, and the PhishGuard audits will run automatically in the background.

### Extension Sub-Systems:
* **Dynamic DOM Scrapers & Wait States:** Uses a `450ms` DOM stabilization wait time to allow Gmail/Outlook's asynchronous dynamic SPAs to complete loading before scraping content.
* **Anchor Fallback Scraper:** Queries Google's internal attachment anchors (`a[href*='view=att']`) to pull attachment filenames, bypassing dynamic classes and fully supporting complex file names containing spaces and parentheses (e.g., `AZTEC_SM_IEEE_LaTeX (1).zip`).
* **Luminance-Based Theme Adaptive Blending:** Traverses DOM parent nodes and calculates background relative luminance. Automatically switches the warnings between Light Mode (`rgba(255, 255, 255, 0.96)`) and Dark Mode (`rgba(30, 41, 59, 0.96)`) glassmorphic CSS systems to blend in seamlessly.
* **Link Redirect Interceptor:** Attaches click listeners on suspicious links inside unsafe emails, prompting warning dialogues before navigation occurs.
* **Safe-Domain Whitelisting:** Local storage system enabling users to whitelist custom domains directly from Gmail banners or the extension popup.

## 📡 API Reference

### `POST /predict`
The main prediction API. Accepts JSON payloads.
```json
{
  "email_text": "From: no-reply@render.com\nSubject: Verify your email\n\nHi, verify your email by clicking the link.",
  "model": "nb"
}
```
Returns a unified hybrid classification block containing predictions, risk scores, DKIM status, attachment metrics, OpenPhish overrides, and XAI feature weights.

### `POST /scan/link`
Audits a standalone URL for IP hosting, brand-spoofing keywords, and length warnings (>150 characters).
```json
{
  "url": "http://192.168.1.1/paypal"
}
```

### `POST /scan/file`
Audits uploaded file signatures. If a `.zip` or `.rar` archive is uploaded, it reads the central directories using memory buffers and reports if there are hidden high-risk executables zipped inside.

## 🧠 Explainable AI (XAI) — Deep Dive

Standard ML models are black boxes. The `explain_prediction()` function in `app.py` opens that box by computing a **per-word importance weight** for every word that appears in both the submitted email and the TF-IDF vocabulary. Only the **top 15 words** by absolute weight are returned.

### How Weights Are Computed Per Model

#### Naïve Bayes (`nb`)
```python
log_prob_diff = clf.feature_log_prob_[1] - clf.feature_log_prob_[0]
weight = log_prob_diff[word_index] × tfidf_value
```

#### Logistic Regression (`lr`)
```python
weight = clf.coef_[0][word_index] × tfidf_value
```

#### Random Forest (`rf`)
```python
direction = 1.0 if word_is_suspicious else -0.5
weight = clf.feature_importances_[word_index] × tfidf_value × direction
```

## 🗄️ Database Schema

Predictions are stored in `predictions.db` (SQLite). The schema is created and **auto-migrated** on every app startup.

```sql
CREATE TABLE IF NOT EXISTS predictions (
    id              INTEGER  PRIMARY KEY AUTOINCREMENT,
    email_text      TEXT     NOT NULL,
    prediction      TEXT     NOT NULL,    -- 'Phishing' or 'Safe'
    confidence      REAL,                 -- 0.0 – 1.0
    timestamp       DATETIME DEFAULT CURRENT_TIMESTAMP,
    model_used      TEXT     DEFAULT 'nb',
    sender          TEXT     DEFAULT 'Unknown',
    risk_score      INTEGER  DEFAULT 0,
    has_url         BOOLEAN  DEFAULT 0,
    attachment_risk TEXT     DEFAULT 'None'
);
```

## 🔒 Security Considerations

| Risk | Mitigation |
|---|---|
| **XSS** | Raw email text is HTML-escaped (`&`, `<`, `>` replaced) before DOM injection |
| **SQL Injection** | All database queries use parameterised `?` placeholders |
| **Code Execution** | Email text is treated as a plain string — it is never `eval()`-ed or executed |
| **Template Injection** | `markupsafe.escape` is imported; Flask's Jinja2 auto-escapes by default |
| **No Dynamic URL Probing** | Uses static analysis of URL strings (IPs, length, typosquatting) to avoid SSRF leaks |
| **Debug Mode** | `debug=True` is disabled in public-facing Render configurations |
