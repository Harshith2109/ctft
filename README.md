# 🛡️ PhishGuard — Explainable AI Phishing Email Detector

A machine-learning-powered web application that detects phishing emails in real time. Built with **Flask**, **scikit-learn**, and **vanilla JavaScript**, it goes beyond a simple "safe/phishing" verdict by providing **explainable AI word-weight heatmaps**, **email header metadata audits** (sender domain spoofing, DKIM, attachments, URLs), a **rule-based risk scoring** system, a **live model performance dashboard**, and a **persistent audit log** — all in a single-page dark-mode UI.

## 🔬 How It Works — The Hybrid Decision Fusion Pipeline

Every email submitted to the application is processed through four simultaneous layers and resolved through a centralized Hybrid Decision logic:

```
                        ┌──────────────────────────────┐
                        │       Email Text Input        │
                        └──────────────┬───────────────┘
                                       │
          ┌────────────────────────────┼────────────────────────────┐
          │                            │                            │
          ▼                            ▼                            ▼
  ┌───────────────┐          ┌──────────────────┐         ┌──────────────────┐
  │ 1. ML Model   │          │ 2. Metadata &    │         │ 3. Rule-Based    │
  │   Classifier  │          │    Threat Intel  │         │    Risk Score    │
  │               │          │                  │         │                  │
  │ TF-IDF Vocab  │          │ SPF/DKIM aligned │         │ Suspicious terms │
  │ vectorization │          │ Attachment ext   │         │ frequency counts │
  │ NB / LR / RF  │          │ OpenPhish Feed   │         │ → 0-100 score    │
  └───────┬───────┘          └────────┬─────────┘         └──────────┬───────┘
          │                           │                              │
          │ (baseline verdict)        │ (trust / threat overrides)   │ (rules data)
          ▼                           │                              │
  ┌───────────────┐                   │                              │
  │ 4. XAI Engine │                   │                              │
  │               │                   │                              │
  │ Computes      │                   │                              │
  │ per-word weight                   │                              │
  └───────┬───────┘                   │                              │
          │                           │                              │
          │ (word weights)            │                              │
          └─────────────┐             │                              │
                        ▼             ▼                              │
                ┌────────────────────────────────┐                   │
                │ 5. Hybrid Decision Logic       │                   │
                │                                │                   │
                │ Applies trust / threat overrides                   │
                │ to compute final threat level  │                   │
                └───────────────┬────────────────┘                   │
                                │                                    │
                                ▼                                    ▼
                  ┌──────────────────────────────────────────────────────┐
                  │                JSON Response to Browser              │
                  │  prediction · confidence · model_used · sender ·     │
                  │  domain_spoof · is_verified_brand · is_dkim_signed · │
                  │  attachment_risk · has_url · suspicious_url ·        │
                  │  risk_score · risk_level · recommended_action ·      │
                  │  xai_weights · stats counters                        │
                  └──────────────────────────────────────────────────────┘
```

### Final Verdict Override Logic

1. **Absolute Trust Override:** If the domain is an authenticated official institutional root (`.gov`, `.gov.in`, `.nic.in`, `.edu`, `.edu.in`, `.ac.in`) or an aligned corporate brand domain, and no spoof is detected, the prediction is **forced to Safe** (risk score capped at 15%) regardless of text suspicion.
2. **Critical Threat Overrides:** The verdict is **forced to Phishing / Dangerous** (risk score forced to >= 80% / 95%) if any of the following threat conditions are met:
   * **Domain Spoofing:** Mismatched senders mimicking well-known brands (typosquatting).
   * **Risky Attachments:** High-risk executables (`.exe`, `.bat`) that haven't been marked as clean by Google Antivirus.
   * **Raw IP Links:** Any URL containing a raw numerical IP address (e.g., `http://192.168.1.1`).
   * **OpenPhish Listed:** Domain is actively listed in the OpenPhish database of active phishing scams.
3. **ML Baseline:** If no overrides trigger, the system defaults to the output of the active TF-IDF machine learning pipeline.

## 🏋️ Training the Model

The model must be trained before the app can make predictions. Run:

```bash
python train_model.py
```

### What the script does step-by-step

| Step | Action |
|---|---|
| **1. Load data** | Reads `phishing.csv`; normalises column names (`email`→`text`, `is_phishing`→`label`) |
| **2. Encode labels** | `LabelEncoder` converts string labels (`"phishing"`, `"safe"`) to integers (`1`, `0`) |
| **3. Split** | 80/20 stratified train/test split (preserves class balance) with `random_state=42` |
| **4. Build pipelines** | Three `sklearn.Pipeline` objects, each: `TfidfVectorizer(stop_words='english', max_features=5000)` → classifier |
| **5. Train classifiers** | MultinomialNB, LogisticRegression (max_iter=1000), RandomForestClassifier (n_estimators=100) |
| **6. Evaluate** | Accuracy, Precision, Recall, F1-score, and Confusion Matrix for each model |
| **7. Save** | `joblib.dump({'models': pipelines, 'metrics': metrics, 'label_encoder': le}, 'model.pkl')` |

### Sample console output

```
Training nb classifier...
[nb] Acc: 0.9823, Prec: 0.9801, Rec: 0.9845, F1: 0.9823
Training lr classifier...
[lr] Acc: 0.9911, Prec: 0.9893, Rec: 0.9930, F1: 0.9911
Training rf classifier...
[rf] Acc: 0.9867, Prec: 0.9852, Rec: 0.9881, F1: 0.9866
Successfully saved all pipelines and metrics to model.pkl
```

### Model Loading in `app.py`

On startup, `app.py` calls `load_model()` which handles three schemas:
- **New schema**: `model.pkl` is a dict with `models`, `metrics`, `label_encoder` keys.
- **Legacy schema**: dict with a `pipeline` key — wrapped into the new format automatically.
- **Bare pipeline**: a single sklearn Pipeline — wrapped with empty metrics.

This ensures backward compatibility if you retrain with a newer version while keeping old `.pkl` files.

## 🧠 Explainable AI (XAI) — Deep Dive

Standard ML models are black boxes. The `explain_prediction()` function in `app.py` opens that box by computing a **per-word importance weight** for every word that appears in both the submitted email and the TF-IDF vocabulary. Only the **top 15 words** by absolute weight are returned.

### How Weights Are Computed Per Model

#### Naïve Bayes (`nb`)

```python
log_prob_diff = clf.feature_log_prob_[1] - clf.feature_log_prob_[0]
weight = log_prob_diff[word_index] × tfidf_value
```

- `feature_log_prob_[1]` = log probability of each word given class = Phishing.
- `feature_log_prob_[0]` = log probability of each word given class = Safe.
- A large **positive** difference → word is much more likely in phishing emails.
- Multiplied by the word's **TF-IDF value** in this specific email to weight by relevance.

#### Logistic Regression (`lr`)

```python
weight = clf.coef_[0][word_index] × tfidf_value
```

- `coef_[0]` = the model's learned coefficients (one per vocabulary feature).
- Positive coefficient → feature contributes to the Phishing class.
- Multiplied by TF-IDF to reflect how prominent the word is in this email.

#### Random Forest (`rf`)

```python
direction = 1.0 if word_is_suspicious else -0.5
weight = clf.feature_importances_[word_index] × tfidf_value × direction
```

- Random Forest doesn't natively provide directional (positive/negative) importance.
- A **heuristic direction** is applied: if the word matches any entry in `SUSPICIOUS_TERMS`, it gets a positive direction (phishing indicator); otherwise negative (safe indicator).
- This is an approximation — RF weights are best interpreted as relative influence, not direction.

### How Weights Are Rendered in the UI

**Heatmap (inline highlighting):**
```
Opacity = min(|weight| × 2.0 + 0.1, 0.6)
Red  background: rgba(239, 68, 68, opacity)   // weight > 0 → Phishing
Green background: rgba(16, 185, 129, opacity)  // weight < 0 → Safe
```

**Feature Table (bar chart):**
```
Bar width % = (|word_weight| / max(|all_weights|)) × 100
```
Each row: word | Phish/Safe Indicator | proportional bar | exact weight (4 d.p.)

## 📧 Email Metadata Analysis

The `parse_email_metadata()` function in `app.py` parses the free-text email body for structured header patterns using regex.

### Header Format (User Input)

```
From: name@domain.com
Signed-by: signingdomain.com
Attachment: filename.exe
```

Any or all of these can be included in the email text. The function is tolerant — if a header is absent, the corresponding field defaults to `Unknown Sender`, empty string, or `None`.

---

### 1. Sender / Domain Analysis

**Parsing:**
```python
re.search(r'(?:from|sender):\s*([^\n\r]+)', text, re.IGNORECASE)
# Then extract email with:
re.search(r'[\w\.-]+@[\w\.-]+\.\w+', sender)
```

**Checks performed on the domain:**

| Check | Logic |
|---|---|
| **Lookalike characters** | Domain contains any of: `paypa1`, `nettflix`, `micr0soft`, `faceb00k`, `secur-update`, `amzn` → `domain_spoof = True` |
| **Brand impersonation** | Domain contains a brand name (e.g. `paypal`) but does NOT match any official domain for that brand → `domain_spoof = True` |
| **Verified brand** | Domain exactly matches or is a subdomain of an official brand domain → `is_verified_brand = True` |

**Brand → Official Domain Mapping:**

| Brand | Official Domains |
|---|---|
| `paypal` | `paypal.com`, `paypal.co.uk` |
| `netflix` | `netflix.com` |
| `google` | `google.com` |
| `gmail` | `gmail.com` |
| `microsoft` | `microsoft.com`, `microsoftonline.com` |
| `outlook` | `outlook.com`, `hotmail.com` |
| `apple` | `apple.com`, `icloud.com` |
| `amazon` | `amazon.com`, `amazon.in` |
| `facebook` | `facebook.com` |

---

### 2. DKIM Verification (Simulated)

**Parsing:**
```python
re.search(r'signed-by:\s*([^\n\r]+)', text, re.IGNORECASE)
```

**Logic:**

| Condition | Result |
|---|---|
| `dkim_domain` matches an official brand root (e.g. `paypal.com` or subdomain) | `is_dkim_signed = True`, `is_verified_brand = True`, `domain_spoof = False` (override) |
| `dkim_domain` present but not in brand list | `is_dkim_signed = False` — shown as unrecognised warning in UI |
| No `Signed-by:` header | `dkim_domain = ""`, `is_dkim_signed = False` |

> This simulates real-world DKIM/SPF alignment: if an email's body claims to be from PayPal AND the signing domain confirms `paypal.com`, the email is treated as legitimate even if the ML model was unsure.

---

### 3. Attachment Risk

**Parsing:**
```python
re.search(r'attachment:\s*([\w\.-]+\.\w+)', text, re.IGNORECASE)
```

**Risk classification:**

| Risk Level | Matched Extensions |
|---|---|
| **High Risk** | `.exe` `.bat` `.scr` `.vbs` `.js` `.cmd` `.ps1` `.lnk` `.zip` `.rar` |
| **Low Risk** | Any other extension (e.g. `.pdf`, `.docx`, `.txt`) |
| **None** | No `Attachment:` header present |

---

### 4. URL Analysis

**Parsing:**
```python
url_pattern = r'https?://[^\s<>"]+|www\.[^\s<>"]+'
urls = re.findall(url_pattern, text)
```

**Flags computed:**

| Flag | Condition |
|---|---|
| `has_url` | `len(urls) > 0` |
| `url_count` | Total number of matched URLs |
| `suspicious_url = True` | Any URL matches bare-IP pattern `http://X.X.X.X/...` |
| `suspicious_url = True` | Any URL has `len(url) > 75` characters |

## 📊 Rule-Based Risk Scoring

The `analyze_text_rules()` function scores every email independently of the ML model using a fixed list of suspicious terms. This provides a human-readable safety signal that works even if the ML model is unavailable.

### Monitored Terms

```python
SUSPICIOUS_TERMS = [
    'urgent', 'verify', 'click here', 'free', 'winner',
    'account suspended', 'claim', 'refund', 'security alert',
    'update card', 'act now', 'limited time'
]
```

> **Intentionally excluded:** `password` and `login` — these appear frequently in legitimate transactional emails (bank statements, streaming service login confirmations) and would generate too many false positives.

### Scoring Formula

```
1. For each term:  count = min(occurrences_in_email, 3)   [cap at 3 to prevent inflation]
2. raw_score = sum(counts)
3. max_possible = 3 × len(SUSPICIOUS_TERMS)  =  3 × 12  =  36
4. score = int((raw_score / max_possible) × 100)
```

### Risk Levels and Actions

| Score | Risk Level | Recommended Action |
|---|---|---|
| 0 – 30 | Low Risk | No action required |
| 31 – 49 | Medium Risk | Verify Sender Identity |
| 50 – 79 | Medium Risk | Mark as Spam & Delete |
| 80 – 100 | High Risk | Delete Email Immediately |

The `found` list (which terms were matched) is also returned but not currently displayed in the UI — available for future enhancement.

## 🔍 Manual Link & File Scan Mechanics

The manual scan dashboard allows forensic analysis of specific links and file uploads using local database caches and static signature inspection.

### 1. Manual Link Scanner (`POST /scan/link`)

When a URL is submitted, the backend analyzes the string structure without triggering outbound HTTP queries (preventing server-side SSRF vulnerabilities):

* **Numerical IP Check:** Detects if the URL hosts a bare IP address (e.g. `http://192.168.1.1/`) which bypasses DNS reputational reviews.
* **Typosquatting & Brand Spoofing:** Checks if the domain contains lookalike brand substrings (such as `paypa1`, `nettflix`, `micr0soft`, `faceb00k`, `amzn`, or `login-`).
* **Length Auditing:** Flags URLs exceeding **150 characters**, which are commonly associated with tracking redirects and hidden token payloads.
* **OpenPhish Verification:** Cross-references the link's domain against the local, 1-hour cached OpenPhish database of active threat URLs.
* **Scoring & Verdict:**
  * **OpenPhish positive:** Instantly overrides the threat level to `High Risk` (risk score of 95%).
  * **Multi-heuristic positive:** Sets the score to `85%` if multiple signs are found, or `65%` if a single sign is found.
  * **Clean scans:** Sets a safe score of `5%`.

---

### 2. Manual File & Archive Scanner (`POST /scan/file`)

Allows binary files to be checked for embedded execution risk before opening. It supports standard files as well as nested archive analysis:

* **Single-File Check:** Audits the uploaded file extension against a predefined dangerous signature blacklist (`.exe`, `.bat`, `.scr`, `.vbs`, `.js`, `.cmd`, `.ps1`, `.lnk`, `.rar`, `.jar`). High-risk matches return `High Risk` (90% risk score).
* **Archive Central Directory Parsing (`.zip`):** 
  When a `.zip` archive is uploaded, the backend reads the archive's **Central Directory** directly from memory buffers (using Python's `zipfile.ZipFile` combined with `io.BytesIO`). It extracts the internal filenames and checks their extensions without unpacking the zip file onto the server storage.
  * If **any** executable file signature is discovered nested inside the zip archive, the entire archive is flagged as a threat and returns a verdict of `High Risk` (80% risk score) with a list of the offending inner paths.
  * If all nested files are low-risk, the scanner lists all nested directories and reports a `Low Risk` (5% risk score).

## 🗄️ Database Schema

Predictions are stored in `predictions.db` (SQLite). The schema is created and **auto-migrated** on every app startup.

```sql
CREATE TABLE IF NOT EXISTS predictions (
    id              INTEGER  PRIMARY KEY AUTOINCREMENT,
    email_text      TEXT     NOT NULL,
    prediction      TEXT     NOT NULL,    -- 'Phishing' or 'Safe'
    confidence      REAL,                 -- 0.0 – 1.0 (NULL if model has no predict_proba)
    timestamp       DATETIME DEFAULT CURRENT_TIMESTAMP,
    model_used      TEXT     DEFAULT 'nb',
    sender          TEXT     DEFAULT 'Unknown',
    risk_score      INTEGER  DEFAULT 0,
    has_url         BOOLEAN  DEFAULT 0,
    attachment_risk TEXT     DEFAULT 'None'
);
```

### Auto-Migration

On startup `init_db()` runs:
```python
c.execute("PRAGMA table_info(predictions)")
columns = [col[1] for col in c.fetchall()]
if 'model_used' not in columns:
    c.execute("ALTER TABLE predictions ADD COLUMN model_used TEXT DEFAULT 'nb'")
# ... same for sender, risk_score, has_url, attachment_risk
```

This means you can update the codebase and restart the app without losing existing data or needing manual SQL migration.

## 🔒 Security Considerations

| Risk | Mitigation |
|---|---|
| **XSS** | Raw email text is HTML-escaped (`&`, `<`, `>` replaced) before injecting into the DOM in `updateResults()` |
| **SQL Injection** | All database queries use parameterised `?` placeholders — no string interpolation in SQL |
| **Code Execution** | Email text is treated as a plain string — it is never `eval()`-ed or executed |
| **Template Injection** | `markupsafe.escape` is imported; Flask's Jinja2 auto-escapes by default |
| **Data Exfiltration** | No network calls are made from the backend — the app is fully offline |
| **Debug Mode** | `debug=True` exposes an interactive debugger; **disable before production** |
